Types of Relationships in Salesforce

Lookup Relationship

  • Real-world: A club can have multiple students enrolled in it.
  • Salesforce: A custom object (Student) can have a lookup relationship to another custom object (Club), where each Student record can hold a reference to a Club.

Features

  • Optional Association: Records in one object can reference records in another object, but this reference is optional.
  • Independent Records: The objects involved in a lookup relationship are not tightly coupled. Deleting one record does not automatically delete the related record.

Example

Clubs
Club ID Club Name
101 Science Club
102 Art Club
103 Music Club
Student
Student ID Student Name Club ID
1 Alice 101
2 Bob 102
3 Charlie 101
4 Diana 103
5 Edward NULL

SOQL Queries

  • Scenario: Query students and the clubs they are associated with.
    • SELECT Id, Name, Club__r.Name FROM Student__c

Master-Detail Relationship

  • Real-world: A course has multiple assignments, and each assignment is tied to a specific course.
  • Salesforce: A Course (Master) and Assignment (Detail), where deleting the Course deletes the associated Assignments.

Features

  • Tightly Coupled Records: The child record (detail) is tightly linked to the parent record (master). The child record cannot exist without the parent.
  • Cascading Deletes: Deleting the parent record also deletes all related child records.
  • Roll-Up Summary Fields: The parent record can have roll-up summary fields that aggregate data from the child records (e.g., count, sum, min, max).
  • Ownership and Sharing: The child record inherits the sharing and security settings of the parent record.

Example

Courses
Course ID Course Name
201 Mathematics
202 History
203 Physics
Assignments
Assignment ID Assignment Name Course ID
301 Algebra Homework 201
302 Geometry Quiz 201
303 WWII Essay 202
304 Physics Lab Report 203
305 Calculus Project 201

SOQL Queries

  • Scenario: Query courses and their related assignments.
    • SELECT Id, Name, (SELECT Id, Name FROM Assignments__r) FROM Course__c

Many-to-Many Relationship

  • Real-world: Students enrolling in multiple courses and courses having multiple students.
  • Salesforce: A junction object (Enrollment) connecting Students and Courses custom objects, allowing multiple students to enroll in multiple courses.

Features

  • Junction Object: A third object (junction object) is used to create a many-to-many relationship between two objects.
  • Bidirectional Relationship: Each record in one object can be related to multiple records in another object, and vice versa.
  • Independent Child Records: Child records in the junction object can reference multiple parent records from both related objects.

  • Scenario: Query courses and the students enrolled in them.
    • SELECT Id, Name, (SELECT Student__r.Id, Student__r.Name FROM Enrollments__r) FROM Course__c

Self Relationship

  • Real-world: A student who can refer other students within the same college.
  • Salesforce: A Student object with a lookup relationship to itself to show mentor-mentee relationships among students.

Features

  • Hierarchical Structure: Allows records within the same object to reference each other, creating a hierarchical relationship.
  • Complex Queries: Supports queries and reports that require hierarchical data structures.
  • Flexibility: Can represent various real-world hierarchical relationships within a single object.

Example

Students
Student ID Student Name Mentor ID
1 Alice 2
2 Bob 3
3 Charlie NULL
4 Diana 1
5 Edward 2

SOQL Queries

  • Scenario: Query students and their mentors (students who referred them).
    • SELECT Id, Name, Mentor__r.Name FROM Student__c

Hierarchical Relationship

  • Real-world: A college organizational chart showing how different departments report to the dean.
  • Salesforce: A special lookup relationship available only for the User object to create a management hierarchy, such as Department Heads reporting to the Dean.

Features

  • User Object Specific: Only available for the User object in Salesforce.
  • Hierarchical Data: Allows for the creation of hierarchical data structures, such as management hierarchies.
  • Reporting and Queries: Supports reporting and queries that require hierarchical data structures, such as organizational charts.

Example

  • Scenario: Query users and their managers.
    • SELECT Id, Name, Manager__r.Name FROM User
